# 서브에이전트 베스트 프랙티스

> 나만의 슬래시 명령어(서브에이전트)를 만들기 위한 원칙과 가이드

---

## 핵심 원칙

### 1. 모듈화 (Modularity)

**"하나의 명령어는 하나의 일만 한다"**

```
❌ 나쁜 예: /do-everything
   → 노트 생성 + 분류 + 링크 + 요약 + 알림...

✅ 좋은 예:
   /daily-note     → 데일리 노트 생성
   /web-clip       → 웹 콘텐츠 저장
   /link-notes     → 노트 연결
```

**왜 모듈화인가?**
- 디버깅이 쉬움 (어디서 문제인지 파악 가능)
- 조합 가능 (필요한 것만 골라 쓰기)
- 수정이 쉬움 (한 부분만 고치면 됨)
- 재사용 가능 (다른 워크플로우에서도 활용)

---

### 2. 단일 책임 (Single Responsibility)

각 명령어는 **명확한 하나의 목적**을 가져야 함

| 명령어 | 책임 |
|--------|------|
| `/daily-note` | 오늘 Daily Note 생성 |
| `/web-clip` | 웹 콘텐츠를 마크다운으로 저장 |
| `/idea-note` | 아이디어 기록 |

**체크리스트:**
- [ ] 이 명령어가 하는 일을 한 문장으로 설명할 수 있는가?
- [ ] 두 가지 이상의 독립적인 작업이 섞여있지 않은가?

---

### 3. 예측 가능성 (Predictability)

**같은 입력 → 같은 결과**

```
/daily-note
→ 항상 50-periodic/51-daily/YYYY-MM-DD.md 에 저장
→ 항상 같은 템플릿 구조
→ 항상 한국 시간 기준
```

**규칙:**
- 저장 위치는 항상 고정
- 파일명 형식은 항상 동일
- 템플릿 구조는 일관되게

---

### 4. 점진적 복잡성 (Progressive Complexity)

**기본은 단순하게, 고급 기능은 선택적으로**

```markdown
## 기본 사용법
/web-clip [URL]

## 고급 사용법 (선택)
/web-clip [URL] --tags "AI,마케팅"
/web-clip [URL] --summary-only
```

## 네이밍 및 번호 규칙

> **참고**: 번호 체계, 폴더 매핑, 템플릿 규칙은 `.claude/CLAUDE.md` 참조

### 명령어명 작성

```
✅ 좋은 예:
/51-daily-note   (번호 + 결과물 중심)
/33-news-briefing (번호 + 동작+대상)

❌ 나쁜 예:
/do-daily        (모호함)
/process         (너무 일반적)
/xyz             (의미 없음)
```

## 확장 가이드

### 새 명령어 만들기

1. **목적 정의**: "이 명령어는 ___를 한다"
2. **저장 위치 결정**: 출처주의 기준 (내가 만든 것? 가져온 것?)
3. **파일명 형식 결정**: `YYYY-MM-DD-제목.md` 등
4. **템플릿 구조 설계**: 프론트매터 + 섹션
5. **작업 흐름 작성**: Step by Step
6. **테스트**: 실제로 실행해보기

### 기존 명령어 수정하기

1. **백업**: 수정 전 원본 복사
2. **최소 변경**: 필요한 부분만 수정
3. **테스트**: 수정 후 동작 확인
4. **문서화**: 변경 사항 기록

## 트러블슈팅

### "명령어가 안 돼요"

1. 파일 위치 확인: `.claude/commands/` 안에 있는지
2. 파일명 확인: `.md` 확장자인지
3. Claude Code 재시작: 터미널에서 `claude` 다시 실행

### "저장 위치가 이상해요"

1. 명령어 파일의 경로 확인
2. CLAUDE.md의 폴더 규칙 확인
3. 상대 경로 vs 절대 경로 확인

### "템플릿이 안 맞아요"

1. 프론트매터 형식 확인 (YAML)
2. 변수 치환 확인 (`{{date}}` 등)
3. 마크다운 문법 확인
